# 外部キーのミニ・アンチパターン

### 標準SQLにおける外部キーの誤った使い方

- **参照方向を逆にしようとする**: FK は子→親の方向で定義する（親テーブルのキーを参照する）
- **作成前のテーブルを参照しようとする**: 参照先テーブルが存在しない状態で FK を定義するとエラー。CREATE TABLE の順序を親→子にする
- **親テーブルのキー以外を参照する**: 参照先は UNIQUE / PRIMARY KEY でなければならない
- **複合キーの列ごとに個別制約を作成しようとする**: 複合 FK は1つの CONSTRAINT でまとめて定義する
- **列の順序が異なる複合 FK を定義しようとする**: 参照元と参照先の列順を揃える
- **データ型の不一致**: 参照元と参照先の型（符号・サイズ）を揃える。INT と INT UNSIGNED も不一致になる
- **文字照合順序の不一致**: 文字列列は CHARACTER SET と COLLATION も一致させる
- **NULL にできない列に SET NULL オプションを使う**: ON DELETE SET NULL は参照元列が NULL 許容でなければエラー
- **重複する制約識別子を作成しようとする**: FK 制約名はスキーマ内でユニークにする
- **循環参照（A→B→A）**: 挿入時に双方の FK を満たせず行き詰まる。PostgreSQL では `DEFERRABLE INITIALLY DEFERRED` でトランザクション終了時まで検証を遅延できる。非対応 DB では一方の FK 列を NULL 許容にして後から UPDATE するか、循環を設計上排除する
- **DROP TABLE 順序の誤り**: FK が存在する状態で親テーブルを先に DROP するとエラー。子テーブルを先に DROP するか、FK 制約を先に DROP する

### MySQLにおける外部キーの誤った使い方

> MySQL 8.0 以降 InnoDB がデフォルトストレージエンジンになっており、以下の多くは新規テーブルでは発生しにくくなっている。レガシーテーブルや明示的に MyISAM を指定している場合に注意。

- **MyISAM テーブルで FK を使おうとする**: MyISAM は FK 非対応（定義しても無視またはエラー）。InnoDB を使う
- **外部キー列に大きなデータ型を使う**: インデックス長の制限に注意（特に長い VARCHAR）
- **UNIQUE でないインデックスへの FK 参照**: MySQL では参照先にインデックスが必要（自動作成されるが意図を明示するため手動定義推奨）
- **インライン参照構文を使う**: MySQL では列定義内の `REFERENCES` は無視される。必ず `CONSTRAINT ... FOREIGN KEY ... REFERENCES` で別途定義する
- **MATCH FULL / MATCH PARTIAL / MATCH SIMPLE を指定する**: MySQL では構文上許可されるが ON DELETE / ON UPDATE が無視されるようになるため使用しない
