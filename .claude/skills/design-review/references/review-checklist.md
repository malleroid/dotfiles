# 設計レビュー 12観点チェックリスト

## 1. 機能の充足性

- 要件・ユースケースに対して機能が過不足なく設計されているか
- エッジケース・例外フローが考慮されているか
- ユーザーストーリーの受入条件を満たせるか

## 2. データモデルの妥当性

- エンティティ間の関係が正しいか（1:N, N:M, 継承等）
- 正規化レベルは適切か（過剰正規化・非正規化の判断根拠）
- データのライフサイクル（作成・更新・削除・アーカイブ）が考慮されているか

## 3. API / インターフェース設計

- 命名規則が一貫しているか
- 入出力の型・制約が明確か
- バージョニング戦略があるか
- エラーレスポンスの設計が統一されているか

## 4. セキュリティ

- 認証・認可の境界が明確か
- 入力バリデーションは適切か（OWASP Top 10 の主要項目）
- 機密データの取り扱い（暗号化・マスキング・ログ除外）
- CORS / CSP / セキュリティヘッダーの考慮

## 5. パフォーマンス・スケーラビリティ

- 想定負荷（TPS, 同時接続数, データ量）に対して設計が耐えうるか
- ボトルネックになりうる箇所の特定と対策
- キャッシュ戦略の妥当性
- 水平・垂直スケーリングの方針

## 6. 信頼性・可用性

- 障害時の影響範囲と復旧手順
- 冗長化・フェイルオーバーの設計
- SLA / SLO の定義と達成可能性
- サーキットブレーカー・リトライ・タイムアウトの設計

## 7. 保守性・可読性

- コンポーネント分割の粒度は適切か
- 責務の分離が明確か（SRP）
- 既存のコードベース・アーキテクチャパターンとの整合性
- 技術的負債の発生リスク

## 8. テスト容易性

- 依存関係の注入・モック化が可能な設計か
- テストの境界（Unit / Integration / E2E）が明確か
- テストデータの準備・クリーンアップが容易か

## 9. 運用・監視性

- ログ出力の設計（構造化ログ、レベル、コンテキスト情報）
- メトリクス・トレーシングの計装ポイント
- アラート条件とエスカレーションパス
- デプロイ・ロールバック手順

## 10. 依存関係・外部連携

- 外部サービスへの依存度と障害時の影響
- API 契約（スキーマ・SLA）の管理方法
- ライブラリ・フレームワークのバージョンとサポート状況
- ベンダーロックインのリスク

## 11. 移行・互換性

- 既存システムからの移行パス
- 後方互換性の維持方針
- データマイグレーションの計画
- 段階的リリース（Feature Flag, Blue-Green, Canary）の考慮

## 12. コスト・制約

- インフラコストの見積もりと最適化
- 開発チームのスキルセットとの適合
- タイムラインに対する設計の実現可能性
- ライセンス・法規制の制約
