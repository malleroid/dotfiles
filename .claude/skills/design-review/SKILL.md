## Reference files

- 12観点チェックリスト: `.claude/skills/design-review/references/review-checklist.md`
- 非機能要件の観点: `.claude/skills/design-review/references/nfr-perspectives.md`

## Your task

Arguments: $ARGUMENTS

### Step 0: 入力の把握

1. Arguments またはこれまでの会話から、レビュー対象の設計内容を特定する
2. 設計ドキュメントのファイルパスが指定されている場合は Read ツールで読み込む
3. レビュー対象が不明な場合は AskUserQuestion で確認する

### Step 1: リファレンスの読み込み

Reference files を Read ツールで読み込む。

### Step 2: コードベースの確認（該当する場合）

設計対象に関連する既存コードがある場合、Glob / Grep / Read で確認し、既存の設計パターン・制約を把握する。

### Step 3: 12観点による評価

`review-checklist.md` の12観点それぞれについて評価する。各観点の評価ステータス:

| ステータス | 意味 |
|-----------|------|
| ✅ | 問題なし・十分に考慮されている |
| ⚠️ | 懸念あり・改善を推奨 |
| ❌ | 重大な問題・対処が必要 |
| ➖ | 今回の設計スコープ外・該当なし |

非機能要件に関する観点（パフォーマンス、セキュリティ、信頼性等）の評価では `nfr-perspectives.md` を参照して深掘りする。

### Step 4: 結果の報告

以下のフォーマットで報告する。

```
## 設計レビュー結果

### サマリー

| # | 観点 | 評価 | 概要 |
|---|------|------|------|
| 1 | 機能の充足性 | ✅/⚠️/❌/➖ | 一言サマリー |
| 2 | データモデルの妥当性 | | |
| ... | ... | | |
| 12 | コスト・制約 | | |

### 総合判定: **APPROVE** / **CONDITIONAL** / **REVISION**

<総合判定の根拠を1-2文で>

### 詳細所見

#### ❌ 重大な問題（対処必須）

- **[観点名]**: <問題の説明>
  - 根拠: <確認済み: ... / 推測: ... / 要検証: ...>
  - 推奨対応: <具体的なアクション>

#### ⚠️ 懸念事項（改善推奨）

- **[観点名]**: <懸念の説明>
  - 根拠: <確認済み: ... / 推測: ... / 要検証: ...>
  - 推奨対応: <具体的なアクション>

#### ✅ 良い点

- **[観点名]**: <評価ポイント>
```

### 判定基準

- **APPROVE**: ❌ が0件、⚠️ が軽微なもののみ
- **CONDITIONAL**: ❌ が0件だが ⚠️ に対処推奨のものあり、または ❌ が1件で軽微
- **REVISION**: ❌ が複数、または重大な ❌ が1件以上

### 反ハルシネーション規則

- 全ての所見に根拠プレフィックスを付与する:
  - **確認済み**: コード・ドキュメント・会話から直接確認した事実
  - **推測**: 一般的なベストプラクティスからの推論
  - **要検証**: 情報が不足しており判断できない項目
- 入力に記載のない要件を勝手に追加しない
- 数値目標が未定義の場合は「未定義（要定義）」と記載する
