# MoSCoW 分類ガイド

要件の優先度を分類するための MoSCoW メソッドのガイドライン。

## 分類定義

| 分類 | 意味 | 判断基準 |
|------|------|---------|
| **Must** | 必須。これがないとリリースできない | ビジネス上の最低要件、法規制対応、安全性に関わるもの |
| **Should** | 重要。可能な限り含めるべき | ユーザー体験に大きく影響、競合優位性に関わるもの |
| **Could** | あると良い。リソースに余裕があれば | UX 改善、利便性向上、nice-to-have 機能 |
| **Won't** | 今回は対象外。将来的に検討 | スコープ外と明確に合意したもの、次フェーズ候補 |

## 分類のガイドライン

### Must の判定

以下のいずれかに該当する場合は Must:
- なければシステムが動作しない
- 法規制・コンプライアンス上必須
- 契約・SLA で約束している
- セキュリティ上の最低要件

### Should の判定

以下のいずれかに該当する場合は Should:
- Must ではないが、ないと運用に支障が出る
- ユーザーからの要望が多い
- 競合製品では標準的に備わっている

### Could の判定

以下のいずれかに該当する場合は Could:
- あればユーザー満足度が上がるが、なくても業務は回る
- 実装コストが低く、余裕があれば入れたい
- 将来的に Should に昇格する可能性がある

### Won't の判定

以下のいずれかに該当する場合は Won't:
- 今回のスコープ外と合意済み
- 技術的・時間的に今回は不可能
- 優先度が低く次フェーズで検討

## 注意点

- **最終判断は人間が行う**: AI は仮分類を提示するが、ビジネス判断は人間に委ねる
- **Must の比率**: 全要件の 60% を超える場合はスコープの見直しを推奨
- **Won't は「やらない」ではなく「今回はやらない」**: 記録として残すことが重要
- **時間軸を明示**: 「今回の Won't」か「当面の Won't」かを区別する
