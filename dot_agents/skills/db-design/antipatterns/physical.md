# 物理設計のアンチパターン

### ラウンディングエラー（丸め誤差）
**目的:** 小数値を格納したい
**アンチパターン:** `FLOAT` / `DOUBLE` / `REAL` データ型を使う
**検出シグナル:** 金額・割合列に FLOAT / DOUBLE / REAL 型
**問題:**
- IEEE 754 の2進浮動小数点では `0.1` を正確に表現できず `0.1 + 0.2 = 0.30000000000000004`
- 等値比較 `WHERE price = 0.1` が意図通りに動かない場合がある
- 誤差は演算のたびに蓄積するため、集計行数が多いほど乖離が拡大する
- Microsoft・PostgreSQL 公式ともに「金融計算・丸め・等値比較には使わないこと」と明記

**解決策:** `NUMERIC(p, s)` / `DECIMAL(p, s)` を使う
- 桁数は通貨・業務要件に合わせて設計する（ISO 4217 で通貨ごとの minor unit が定義されている）
- SQL Server エコシステムでは `DECIMAL(19, 4)`（`MONEY` 型相当）が慣習的に使われる

**例外:**
- 科学計算など近似値で十分な用途
- 地理座標: PostGIS / MySQL POINT 型など DB 専用の空間型を使う場合は内部的に float8 が使われるため別論点

### サーティワンフレーバー（31のフレーバー）
**目的:** 列を特定の値に限定したい
**アンチパターン:** `ENUM('active', 'inactive', 'pending')` のように列定義に値を埋め込む
**検出シグナル:** ENUM 型の列、CHECK 制約に値のリストを直書き
**問題:**
- 値の追加・変更・削除にスキーマ変更が必要
- MySQL 9: 値をリスト途中に追加すると全テーブルロック＆書き直しが発生
- PostgreSQL（12 以降）: 値の追加・リネームはロックなしで可能だが、削除・順序変更は型の DROP & 再作成が必要
- SQL 標準ではなく移植性が低い
- アプリ側 Enum との二重管理で同期ずれが起きやすい

**解決策（3択）:**

| 方式 | 変更のしやすさ | パフォーマンス | 向いている場面 |
|------|--------------|--------------|--------------|
| ルックアップテーブル + FK | ◎ INSERT/DELETE のみ | △ FK チェックのオーバーヘッド | 値が変わりうる、説明文・メタデータが必要 |
| CHECK 制約 + VARCHAR | ○ DROP & ADD が必要 | ○ | 値が少なく変更頻度が低い、移植性重視 |
| ENUM | △ スキーマ変更必須 | ◎ | 値が固定で変更不要、単一 DB エンジン限定 |

**例外:** 値が仕様上絶対に変わらないと確信できる場合のみ ENUM を検討する

### ファントムファイル（幻のファイル）
**目的:** 画像などのバイナリファイルを扱いたい
**アンチパターン:** ファイルをファイルシステムに保存し、DB にはパスだけ格納する
**検出シグナル:** `file_path VARCHAR` 列、`image_url VARCHAR` 列でファイル実体が DB 外にある
**問題:**
- DB レコード削除時にファイルが残る（ファントムファイル）、またはファイル削除時に DB が孤立レコードを持つ
- トランザクションのロールバックでファイルは元に戻らない
- バックアップ・リストア時に DB とファイルの整合が取りにくい

**解決策（3択）:**

| 方式 | 整合性 | コスト・スケール | 向いている場面 |
|------|--------|----------------|--------------|
| オブジェクトストレージ（S3等）+ DB にキー格納 | △ アプリ側で管理 | ◎ 安価・無制限 | 現代 Web アプリの標準。大容量・CDN 配信 |
| DB の BLOB 型 | ◎ トランザクション保証 | △ DB が肥大化・バックアップ増大 | 機密ファイル・小容量・強整合性が必須 |
| ファイルシステム + パス格納 | △ アプリ側で管理 | ○ | オンプレ・CDN 不要の小規模システム |

**注意:**
- オブジェクトストレージ利用時は削除・更新のロジックをアプリ側で整合させる（Soft Delete パターン等）
- PostgreSQL の Large Object（`pg_largeobject`）は行削除しても自動削除されないため、定期パージが必要

### インデックスショットガン（闇雲インデックス）
**目的:** クエリパフォーマンスを最適化したい
**アンチパターン:** インデックスを作らない、または闇雲に大量作成する
**検出シグナル:**
- インデックスなしの大テーブル（`pg_stat_user_tables.seq_scan` が多い）
- 全列または大半の列にインデックスがある
- 未使用インデックス（`pg_stat_user_indexes.idx_scan = 0` が続く）

**問題:**
- 未作成 → フルスキャンによるクエリ遅延
- 過剰作成 → INSERT/UPDATE/DELETE のオーバーヘッド、VACUUM コスト増大、ストレージ肥大
- 未使用インデックスも VACUUM の対象となり I/O を消費し続ける

**解決策:** **MENTOR** の原則に従う
- **M**easure（測定）: クエリプロファイリングで遅いクエリを特定
- **E**xplain（解析）: `EXPLAIN (ANALYZE, BUFFERS)` で実行計画を確認
- **N**ominate（指名）: 候補インデックスを選定（複合インデックスは選択性の高い列を先頭に）
- **T**est（テスト）: 効果を計測（本番相当のデータ量で検証）
- **O**ptimize（最適化）: カバリングインデックス（`INCLUDE`）・部分インデックスを検討
- **R**ebuild（再構築）: 未使用インデックスを削除、断片化したインデックスを再構築

**インデックス種類の選択（PostgreSQL）:**

| 種類 | 向いている用途 |
|------|--------------|
| B-tree（デフォルト）| 等値・範囲・ソート |
| GIN | 配列・JSONB・全文検索 |
| BRIN | 時系列など物理的に連続した大テーブル |
| GiST | 地理データ・範囲型 |
