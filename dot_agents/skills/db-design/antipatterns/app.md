# アプリケーション開発のアンチパターン

### リーダブルパスワード（読み取り可能パスワード）
**目的:** パスワードのリカバリーとリセットを行いたい
**アンチパターン:** パスワードを平文・Base64・可逆暗号化（AES等）で格納する
**検出シグナル:**
- `password VARCHAR` 列に平文・Base64・暗号化値が格納されている
- パスワードをメール本文で送信している（平文保持の証拠）
- 「パスワードを忘れた」でパスワードそのものを返送できる機能がある

**問題:**
- DB 漏洩時に全パスワードが即時露出する
- 可逆暗号化は暗号鍵が漏洩した瞬間に平文保存と等価になる
- ユーザーは複数サービスでパスワードを使い回す傾向があり、1サービスの漏洩が他サービスへの不正ログイン（クレデンシャルスタッフィング）に連鎖する
- OWASP Top 10:2025 の A04（暗号の失敗）に分類される

**解決策:**
- パスワードは必ず一方向ハッシュ関数で格納する（復元不可能であることが正しい設計）
- アルゴリズムは以下の優先順位で選択する（OWASP 推奨）:
  1. **Argon2id**（最推奨）: メモリハード関数。GPU/ASIC による総当たりを困難にする
  2. **scrypt**（Argon2id が使えない場合）
  3. **bcrypt**（レガシーシステムのみ）: ワークファクター 10 以上
  4. **PBKDF2-HMAC-SHA256**（FIPS-140 準拠が必須の場合のみ）
- **ペッパー**: DB 外（環境変数・シークレット管理サービス）に保管する共通シークレットをハッシュ前に結合することで、DB 単独漏洩時の総当たりを防ぐ
- パスワードはリセットフローのみ提供し、リカバリー（元の値の復元）は実装しない

### SQLインジェクション
**目的:** 動的な SQL クエリを組み立てたい
**アンチパターン:** ユーザー入力を文字列結合して SQL に埋め込む
**検出シグナル:**
- `"SELECT ... WHERE id = " + userId` のような文字列結合
- `f"SELECT ... WHERE name = '{name}'"` のような文字列補間
- `execute("... WHERE col = '" + val + "'")` のような直接実行

**問題:**
- OWASP Top 10 の常連であり、最大級の Web セキュリティ脅威
- DB の全データ取得・削除・改ざん、認証バイパス、OS コマンド実行が可能
- 攻撃パターンが広く知られており、自動スキャンツールで容易に発見される

**解決策（優先順位順）:**

1. **プレースホルダ・バインド変数**（最優先）: SQL コードとデータを分離して渡す。DB はコードとデータを常に区別するため注入が不可能
   ```python
   # NG
   cursor.execute(f"SELECT * FROM users WHERE id = {user_id}")
   # OK
   cursor.execute("SELECT * FROM users WHERE id = ?", (user_id,))
   ```
2. **ORM の使用**: パラメータ化クエリを自動的に生成する。生の SQL を書く機会を減らす
3. **許可リスト検証**: テーブル名・列名など、バインド変数が使えない動的箇所のみに適用
4. **エスケープ処理**: 最後の手段。OWASP が「強く非推奨」と明記。上記が使えない場合に限定する

**追加防御（多層防御として併用）:**
- 最小権限の DB アカウント（アプリ用アカウントに DBA 権限を与えない）
- WAF（Web Application Firewall）による既知パターンのブロック
- エラーメッセージに DB の詳細（テーブル名・列名）を含めない

### シュードキー・ニートフリーク（疑似キー潔癖症）
**目的:** 連番 ID の欠番を詰めたい
**アンチパターン:** 削除後に ID を振り直したり、欠番を再利用する
**検出シグナル:**
- 削除後に `UPDATE ... SET id = id - 1 WHERE id > ?` のような ID の詰め直し
- 最大 ID を取得して +1 する独自採番ロジック（シーケンス・SERIAL を使わない）
- 「欠番があるのはバグでは？」というコメントや Issue

**問題:**
- 外部キーや他テーブルで参照中の ID を変更すると参照整合性が破壊される
- ブックマーク URL・API エンドポイント・外部システムの連携キーが無効になる
- 再採番中に別トランザクションが同じ ID を参照する競合状態が発生する
- 連番 ID はそもそも推測が容易なため、欠番の有無に関わらず IDOR（Insecure Direct Object Reference）対策は別途必要

**解決策:**
- 疑似キーの欠番は埋めない。連番は「ユニークな識別子」であり「順序番号」ではない
- 採番は DB のシーケンス機能（SERIAL / IDENTITY / SEQUENCE）に委ねる
- IDOR リスクが気になる場合は ID 型を UUID/ULID 等に変更する（logical.md「IDリクワイアド」参照）

**例外:** 表示用の通し番号が必要な場合は `ROW_NUMBER()` ウィンドウ関数で別途生成し、永続化しない

### シー・ノー・エビル（臭いものに蓋）
**目的:** 簡潔なコードを書きたい
**アンチパターン:** DB のエラー・戻り値・例外を無視する
**検出シグナル:**
- `execute()` / `query()` の戻り値（影響行数）を捨てている
- 空の catch/except ブロック（`except: pass`、`catch (e) {}`）
- エラー発生時に ROLLBACK せずに処理を続行している

**問題:**
- DB 操作が失敗しても上位層に伝播せず、静かに失敗する（Silent Failure）
- 更新が 0 行で終わったことに気づかず、データが意図通りに変更されていない状態が続く
- トランザクションの途中でエラーが発生しても ROLLBACK されず、部分的な更新状態が残る
- 不整合は発生から発覚まで時間がかかるほど修正コストが増大する

**解決策:**
- 戻り値と例外を必ずチェックする
  - `execute()` の戻り値（影響行数）が期待値と異なる場合はエラーとして扱う
  - 例外は握りつぶさず、ログに記録して上位に伝播させる
- エラー発生時は必ず ROLLBACK し、中途半端な状態を残さない
- DB 操作のエラーログにはクエリパラメータ（機密情報を除く）・エラーコード・スタックトレースを含め、再現・デバッグを容易にする
- 開発環境では実際に実行された SQL を出力して意図通りかを確認する（ORM のクエリログ等）

### ディプロマティック・イミュニティ（外交特権）
**目的:** ベストプラクティスを採用したい
**アンチパターン:** SQL・スキーマをソフトウェアエンジニアリングの原則の例外扱いにする
**検出シグナル:**
- DDL が Slack・チケット・Wiki にしか存在せず、リポジトリで管理されていない
- 本番 DB への直接 ALTER TABLE（マイグレーションファイルが存在しない）
- スキーマ変更が PR レビューを経ずにデプロイされる

**問題:**
- スキーマの変更履歴が追えず、いつ・誰が・なぜ変更したかが不明になる
- ロールバック手順が存在しないため、本番障害時の復旧コストが跳ね上がる
- 環境間（開発・ステージング・本番）でスキーマが乖離し、「本番だけで起きるバグ」を生む
- テストされていないスキーマ変更が本番でのみ発覚するリスクがある

**解決策:**
- **DDL のバージョン管理**: マイグレーションファイル（Flyway / Liquibase / Atlas 等）でスキーマ変更を追跡する
- **アプリコードと同一リポジトリ**: スキーマ変更とアプリ変更を1コミットにまとめることで、機能単位での原子的なデプロイが可能になる
- **スキーマ変更のコードレビュー**: DDL も PR レビューを必須とし、SQL lint・命名規約チェックを CI で自動化する
- **本番同等データでのテスト**: 空のスキーマではなく本番に近いデータでマイグレーションを検証する
- **DB レベルのテスト**: データ整合性・制約・インデックスの効果をテストする

### スタンダード・オペレーティング・プロシージャ（さびついた開発標準）
**目的:** ストアドプロシージャを活用したい
**アンチパターン:** ビジネスロジックをすべてストアドプロシージャに実装する
**検出シグナル:**
- 「ビジネスロジックは全部ストアドプロシージャに」という方針がある
- アプリコードが DB への呼び出しだけになっており、ロジックが SP 側に集中している
- SP が 1000 行超で複数の責務を持っている

**問題:**
- **テスト困難**: ユニットテストに DB 接続が必要で、依存のモック・独立したテストが難しい
- **デバッグ困難**: ステップ実行・ブレークポイント・ログ出力のエコシステムがアプリ言語より貧弱
- **バージョン管理の分断**: SP の変更が Git フローに乗りにくく、アプリコードとのセットデプロイが難しくなる（ディプロマティック・イミュニティ参照）
- **スケーリング困難**: SP は DB サーバー上で実行されるため、処理を水平スケールできない
- **開発者・DBA のサイロ化**: ロジックが分散し、重複実装や不整合が起きやすくなる

**解決策:**
- ビジネスロジックはアプリ層に持つ。テスト・デバッグ・デプロイの成熟したエコシステムを活用する
- ストアドプロシージャが有効な限定的な場面:
  - **大量データの集計・バッチ処理**: ネットワーク越しのデータ転送を避けたい場合
  - **ORM では表現しにくい複雑なクエリ**: ウィンドウ関数・再帰 CTE 等の SQL の表現力を活かす場合
  - **DB 側でのアクセス制御**: SP 経由のみテーブルアクセスを許可するセキュリティ要件がある場合
- SP を使う場合も、マイグレーションファイルで管理し PR レビューを必須とする
