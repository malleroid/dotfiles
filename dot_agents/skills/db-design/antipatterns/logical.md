# 論理設計のアンチパターン

### ジェイウォーク（信号無視）
**目的:** 複数の値を持つ属性を1列に格納したい
**アンチパターン:** カンマ区切りリストを VARCHAR に格納する
**検出シグナル:**
- `LIKE '%,value,%'` による部分一致検索
- 列コメントに「カンマ区切り」「複数ID」の記述
- `tags VARCHAR(255)` のような列名と型の組み合わせ

**問題:**
- 特定値の検索・更新・削除が困難（インデックスが効かない）
- 集約・カウントにアプリ側パース処理が必要
- 参照整合性（FK）を宣言できない
- 区切り文字を含む値が格納できない

**解決策（3択）:**

| 方式 | 整合性 | クエリのしやすさ | 向いている場面 |
|------|--------|----------------|--------------|
| 交差テーブル（中間テーブル） | ◎ FK 制約可 | ◎ JOIN で自在 | 参照整合性が必要、値にメタデータがある |
| DB ネイティブの配列・リスト型 | △ FK 不可 | ○ 専用演算子で検索可 | タグ・ラベルなど整合性不要の単純リスト |
| JSON / 半構造化型 | △ FK 不可 | △ JSON 演算子で検索可 | 構造が不定・外部連携データの一時格納 |

**例外:** 読み取り専用の外部データ取り込み、ログ等で参照整合性が不要な場合

---

### ナイーブツリー（素朴な木）
**目的:** 階層構造（ツリー）を格納・クエリしたい
**アンチパターン:** 隣接リスト（adjacency list）を使いつつ、アプリ側のループで階層を走査する
**検出シグナル:**
- `parent_id` 列のみで階層を表現
- 子孫取得のために N+1 クエリを発行しているアプリコード

**問題:**
- 任意深さの祖先・子孫を1クエリで取得できない
- アプリ側ループによる N+1 クエリでパフォーマンスが劣化
- 深さが増えるほど影響が拡大する

**解決策（4択）:**

| 方式 | 読み取り | 書き込み | 向いている場面 |
|------|--------|--------|--------------|
| 隣接リスト + WITH RECURSIVE | ○ | ◎ | 標準 SQL 対応 DB、変更が多いツリー |
| 閉包テーブル | ◎ | △ 全パスの更新が必要 | 読み取り頻度が高い、サブツリー操作が多い |
| 経路列挙（Materialized Path）| ○ 前方一致で検索可 | ○ | シンプルな実装優先、ファイルパス的な用途 |
| 入れ子集合（Nested Set）| ◎ | ✕ 再番号付けが必要 | 読み取り専用に近い静的ツリー |

**例外:** 2階層以下の浅い固定階層では隣接リストで十分

---

### IDリクワイアド（とりあえずID）
**目的:** 主キーの規約を統一したい
**アンチパターン:** すべてのテーブルに `id` という名の連番列を無条件に追加する
**検出シグナル:**
- 自然キー（メールアドレス・国コード等）が存在するのに `id` を追加している
- 交差テーブル（多対多の中間テーブル）に不要な `id` がある
- 複合キーが適切な場面で単一の `id` 列に置き換えている

**問題:**
- 自然キーや複合キーに UNIQUE 制約がなければ重複行を防げない
- 結合クエリで `id` が複数テーブルから登場し、列名の意味が不明確になる
- 連番整数は値の推測が容易で IDOR（Insecure Direct Object Reference）のリスクがある

**解決策:**
- 自然キーや複合キーが安定している場合はそちらを PRIMARY KEY に使う
- 代理キーを使う場合は `account_id` のような意味ある列名にする
- 代理キー + 自然キーの UNIQUE 制約を併用するハイブリッドアプローチが実用的
- 代理キーの型は要件に応じて選択する：

| 型 | 特性 | 向いている場面 |
|----|------|--------------|
| 連番整数（SERIAL / IDENTITY） | 高速・小サイズ・予測可能 | 内部結合のみ、公開しない ID |
| UUID（v4） | 分散生成可・衝突なし・ランダム | 分散環境・公開 API |
| ULID / UUIDv7 / TSID | 時系列ソート可・分散生成可 | 分散環境 + インデックス効率を両立したい場合 |

**例外:** ORM のデフォルト規約（Rails 等）に従う場合は `id` が強制されるが、UNIQUE 制約による重複防止は別途必要

### キーレスエントリ（外部キー嫌い）
**目的:** DB アーキテクチャを単純化したい
**アンチパターン:** 外部キー制約を使用しない
**検出シグナル:**
- `_id` 列があるのに FOREIGN KEY 宣言なし
- 孤立レコード（参照先が存在しない行）が発生している
- 「FK はパフォーマンスに悪い」という理由で一律無効化している

**問題:**
- アプリコードのバグ・直接 SQL 操作・マイグレーション漏れなどで孤立レコードが無音で蓄積する
- データ不整合は発生から発覚まで時間がかかり、修正コストが増大する
- FK がない場合、JOIN 結果の欠損がバグではなく「仕様」に見えてしまう

**解決策:**
- 外部キー制約を宣言し、DB レベルで整合性を保証する
- `ON DELETE` / `ON UPDATE` の挙動を明示的に指定する（デフォルトの RESTRICT が安全）
- `ON DELETE CASCADE` は便利だが、意図せず大量削除が連鎖するリスクを理解して使う

**例外（FK を使わない正当な理由）:**
- マイクロサービスの DB-per-service パターン: サービス境界をまたぐ参照は FK で表現できない。代わりにイベント駆動や API による結果整合性で管理する
- シャーディング・パーティションをまたぐ参照
- 大量バルクロード時の一時的無効化（ロード後に再有効化すること）

---

### EAV（エンティティ・アトリビュート・バリュー）
**目的:** 可変属性（スキーマレスな属性）をサポートしたい
**アンチパターン:** `(entity_id, attribute_name, value)` の汎用属性テーブルを使う
**検出シグナル:**
- `attribute_name` / `attr_key` / `property_name` 列と TEXT 型の `value` 列の組み合わせ
- 1エンティティの全属性を取得するために多数の行を JOIN・ピボットしている

**問題:**
- `value` が TEXT 型のため数値・日付などのデータ型を保証できない
- NOT NULL・CHECK などの制約を属性ごとに宣言できない
- 1エンティティのすべての属性を1行に再構成するにはピボット（PIVOT / GROUP BY + 集約）が必要でクエリが複雑化する
- インデックスが効きにくく、大量データでパフォーマンスが劣化する

**解決策（属性の性質で選択）:**

属性がある程度固定・予測可能な場合:

| 方式 | 向いている場面 |
|------|--------------|
| シングルテーブル継承 | サブタイプが少なく NULL 列が許容できる |
| 具象テーブル継承 | サブタイプごとに明確に分離したい |
| クラステーブル継承 | 共通属性と差分を分けて管理したい |

属性が真に動的・ユーザー定義の場合:
- **JSON / 半構造化型**を使う。EAV に比べてクエリが簡潔で、インデックスも利用可能

**例外:** EAV が正当化されるケースはほぼない。動的属性が必要な場合は JSON / 半構造化型で対処する

---

### ポリモーフィック関連
**目的:** 1つのテーブルで複数の親テーブルを参照したい
**アンチパターン:** `parent_type` + `parent_id` という二重目的の列で複数の親を参照する
**検出シグナル:**
- `commentable_type` / `taggable_type` のような `*_type` 列と `*_id` 列のペア
- `_type` 列の値がテーブル名やクラス名の文字列

**問題:**
- `parent_id` がどのテーブルを参照するか型によって変わるため、FK 制約を宣言できない
- 孤立レコードや型の不整合を DB レベルで防げない
- JOIN に CASE 式や動的 SQL が必要になりクエリが複雑化する

**解決策:**

| 方式 | 概要 | 向いている場面 |
|------|------|--------------|
| 排他的弧（Exclusive Arc） | 親テーブルごとに個別の FK 列を持ち、CHECK 制約で1つだけ非 NULL を強制 | 親テーブルの種類が少なく固定 |
| 共通基底テーブル | 親テーブルを共通の基底テーブルのサブタイプにし、FK はその基底テーブルへ | 親テーブル間に共通属性がある |
| 参照を逆向きにする | 各親テーブル側に子テーブルへの関連テーブルを持つ | 親テーブルが多い・拡張が見込まれる |

**例外:** ORM の polymorphic 機能（Rails 等）を使う場合は DB レベルの整合性保証がないことを認識し、定期的な孤立レコード検査・アプリ側での型バリデーションを必ず実施する

---

### マルチカラムアトリビュート（複数列属性）
**目的:** 複数の値を持つ属性を格納したい
**アンチパターン:** `tag1`, `tag2`, `tag3` のように連番の列を複数定義する
**検出シグナル:**
- 連番サフィックスの列（`phone1` / `phone2`、`tag1` / `tag2` / `tag3`）
- NULL 許容の同種列が複数並ぶ
- `WHERE tag1 = ? OR tag2 = ? OR tag3 = ?` のような OR を連ねた検索クエリ

**問題:**
- 第1正規形（1NF）違反: 繰り返しグループを列で表現している
- N 番目の値を検索・更新するクエリが列数分の OR または CASE 式を必要とする
- 値を追加するたびに ALTER TABLE で列を増やす必要がある
- 使われていない列が NULL で埋まり、テーブルが横方向に肥大化する

**解決策（3択）:**

| 方式 | 向いている場面 |
|------|--------------|
| 従属テーブル（子テーブル）+ FK | 参照整合性が必要、値にメタデータがある |
| DB ネイティブの配列・リスト型 | 順序なし・整合性不要の単純リスト |
| JSON / 半構造化型 | 構造が不定・外部連携データ |

**例外:** 固定数・固定順序の意味ある属性（例: 座標の `x` / `y` / `z`、色の `r` / `g` / `b`）

---

### メタデータトリブル（メタデータ大増殖）
**目的:** データ量増加に対してスケーラビリティを高めたい
**アンチパターン:** 年別・月別などでテーブルや列を手動でコピー増殖させる
**検出シグナル:**
- `orders_2023` / `orders_2024` のような日付サフィックス付きテーブル群
- `sales_jan` / `sales_feb` のような期間サフィックス付き列群
- 集計クエリが `UNION ALL` でテーブルを連結している

**問題:**
- 複数テーブルをまたぐ集計・検索に `UNION ALL` が必要でクエリが複雑化する
- テーブルごとに FK・制約・インデックスを個別管理する必要がある
- 新しい期間が来るたびに DDL 作業が発生し、マイグレーションが困難になる
- テーブル数の爆増により DB オブジェクト管理が破綻する

**解決策:** DB ネイティブのパーティショニング機能を使う

| パーティション種別 | 向いている場面 |
|------------------|--------------|
| RANGE（範囲）| 日付・連番など連続値での分割（最も一般的） |
| LIST（リスト）| 国・地域・カテゴリなど離散値での分割 |
| HASH | 特定のキーなし・均等分散させたい場合 |

パーティショニングによりアプリ側のクエリは単一テーブルのまま、DB が自動でパーティションプルーニングを行う。

**例外:** DB ネイティブのパーティショニングが使えない環境での暫定策、または法規制によるデータ保持期間管理でアーカイブテーブルを明示的に分離する場合

---

## 付録A　正規化のルール（要点）

| 正規形 | 条件 |
|--------|------|
| 第1正規形（1NF） | 各列は単一の原子値。繰り返しグループなし |
| 第2正規形（2NF） | 1NF かつ、非キー列が複合主キー全体に関数従属 |
| 第3正規形（3NF） | 2NF かつ、非キー列間の推移的関数従属を排除 |
| ボイス・コッド正規形（BCNF） | すべての関数従属の決定子がスーパーキー |
| 第4正規形（4NF） | BCNF かつ、多値従属性を排除 |

**実用的な目安:** 通常は 3NF / BCNF を目標とする。意図的な非正規化はパフォーマンス要件を文書化した上で行う。
